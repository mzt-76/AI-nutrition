### Issue #1

**Severity:** high
**Category:** Logic
**File:** `frontend/src/pages/MealPlanView.tsx:14-25`
**Verified:** verified against `skills/meal-planning/scripts/generate_day_plan.py:106-118`

**Issue:**
`ingredients` type mismatch — frontend expects `string[]`, backend stores `Array<{name, quantity, unit}>`.

**Detail:**
`generate_day_plan.py` line 109 passes through `scaled_recipe.get("ingredients", [])` which are structured objects from the recipe DB (`{name: "Poulet", quantity: 200, unit: "g"}`). `MealPlanView.tsx` and `MealCard.tsx` treat them as plain strings and call `.join(', ')`. This will render `[object Object], [object Object]` in the browser.

**Current Code:**
```typescript
// MealPlanView.tsx:143
ingredients: m.ingredients,

// MealCard.tsx:39
<p className="text-xs text-gray-400">{ingredients.join(', ')}</p>
```

**Suggested Fix:**
Transform in `MealPlanView.tsx` when mapping, and update types + Zod schema to accept both formats. `MealCard` stays as-is (receives clean `string[]`).

### Issue #4

**Severity:** medium
**Category:** Quality
**File:** `frontend/src/lib/api.ts:10`
**Verified:** probable

**Issue:**
`new URL(AGENT_ENDPOINT).origin` throws at import time if env var is missing or relative path.

**Detail:**
If `VITE_AGENT_ENDPOINT` is unset or set to a relative path like `/api/agent`, `new URL()` throws `TypeError: Invalid URL`. This crashes the entire app at module load. The original `AGENT_ENDPOINT` constant was fault-tolerant (just `undefined`).

**Suggested Fix:**
Use a try/catch fallback:
```typescript
export const API_BASE_URL = (() => {
  try { return new URL(AGENT_ENDPOINT).origin; }
  catch { return ''; }
})();
```
