### Issue #3

**Severity:** high
**Category:** Logic
**File:** `frontend/src/components/tracking/PlanValidation.tsx:10-15` and `frontend/src/hooks/useDailyTracking.ts:218-223`
**Verified:** verified — confirmed by reading `skills/meal-planning/SKILL.md:187` and `generate_day_plan.py:107`

**Issue:**
`BADGE_COLORS` key lookup and `mealTypeMap` in `logPlanMeal` both rely on `meal.meal_type` being a French display string (e.g. `"Petit-déjeuner"`), but the backend contract stores the **raw display name from the meal slot template** which is always that exact capitalised French form. This works today but is fragile and already diverges internally.

**Detail:**
`generate_day_plan.py:107` passes `meal_slot.get("meal_type", "Repas")` verbatim as the `meal_type` field in the stored plan. The slot templates use `"Petit-déjeuner"`, `"Déjeuner"`, `"Dîner"`, `"Collation"`. `useDailyTracking.ts:218-223` maps these back to DB keys using a hardcoded dict. If a meal slot ever has a non-standard display name (e.g. `"Collation AM"` — which the SKILL.md explicitly lists as a valid slot in the `3_meals_2_snacks` template), the map falls back to `'collation'` silently with no warning. More critically, `BADGE_COLORS` in `PlanValidation.tsx` will miss it and render `bg-gray-500/15` instead of the expected colour.

**Current Code:**
```typescript
// useDailyTracking.ts:218-223
const mealTypeMap: Record<string, MealType> = {
  'Petit-déjeuner': 'petit-dejeuner',
  Déjeuner: 'dejeuner',
  Dîner: 'diner',
  Collation: 'collation',
};
const entry = { meal_type: mealTypeMap[meal.meal_type] ?? 'collation', ... };
```

**Suggested Fix:**
The backend should normalise `meal_type` at storage time (in the `_build_meal_from_recipe` function in `generate_day_plan.py`), storing the DB key slug (`petit-dejeuner` etc.) rather than the display string. Until then, make the frontend map case-insensitive and substring-tolerant, or add a console warning on unknown values.

### Issue #4

**Severity:** medium
**Category:** Logic / UX
**File:** `frontend/src/hooks/useDailyTracking.ts:251-255`
**Verified:** probable

**Issue:**
`isPlanMealLogged` uses `food_name` equality to detect duplicates, which will falsely flag a meal as already logged if the same recipe name appears in a different plan or on a different date.

**Detail:**
The check is:
```typescript
entries.some((e) => e.meal_plan_id === activePlanId && e.food_name === meal.name)
```
This does not filter by `log_date`. Since `entries` is fetched for the current `dateStr`, the date is implicitly scoped correctly via `refreshEntries`. However, the match on `food_name` alone (without a per-day meal-type check) means that if the user logs "Salade niçoise" at lunch and the plan also has "Salade niçoise" as a dinner option, the dinner row will appear as already logged. A more precise check would also compare `meal_type`.

**Suggested Fix:**
```typescript
const isPlanMealLogged = useCallback(
  (meal: PlanMeal): boolean => {
    const dbKey = mealTypeMap[meal.meal_type] ?? 'collation';
    return entries.some(
      (e) =>
        e.meal_plan_id === activePlanId &&
        e.food_name === meal.name &&
        e.meal_type === dbKey,
    );
  },
  [entries, activePlanId],
);
```
(Requires extracting `mealTypeMap` to module scope.)

### Issue #5

**Severity:** medium
**Category:** Logic / Reliability
**File:** `frontend/src/components/tracking/TrackingInput.tsx:52`
**Verified:** probable

**Issue:**
`setTimeout(() => onEntryCreated(), 1000)` is an arbitrary race condition guard that will silently fail if the backend is slow.

**Detail:**
After calling `sendMessage`, the component waits 1 second then triggers a DB refresh. This assumes the agent will have written the log entry to Supabase within one second. If the agent is under load or the LLM takes longer, the refresh will run before the row is committed and the user will see no new entry — with a "Enregistré" success toast already shown. The user has no way to know the display is stale.

**Suggested Fix:**
Either (a) poll with a short retry until a new entry appears, or (b) change the API endpoint to return the created log entry IDs so the frontend can confirm them. A simple improvement with current architecture: increase the timeout to 2500ms as a partial mitigation, and add a second refresh on component focus/visibility.

### Issue #7

**Severity:** medium
**Category:** Accessibility
**File:** `frontend/src/components/tracking/DateSelector.tsx:17-51`, `frontend/src/components/tracking/MealSection.tsx:68-76`, `frontend/src/components/tracking/TrackingInput.tsx:77-89`, `frontend/src/components/tracking/PlanValidation.tsx:40-81`
**Verified:** verified — `grep -rn "aria-" tracking/` returns zero results across all 6 components

**Issue:**
All interactive controls across every tracking component are missing `aria-label` attributes. Screen readers cannot describe any action.

**Detail:**
Specific gaps:
- `DateSelector`: Previous/Next `<Button>` icons render only a `ChevronLeft`/`ChevronRight` SVG with no label. The "Revenir à aujourd'hui" `<button>` has visible text but no `role` or `aria-*` attributes. The chevron buttons are icon-only — a screen reader announces them as unlabelled buttons.
- `MealSection`: The delete `<Button>` renders only a `Trash2` icon with no `aria-label`. A screen reader cannot tell which food item will be deleted.
- `TrackingInput`: The microphone toggle button has no `aria-label` and no `aria-pressed` state.
- `PlanValidation`: Plan meal rows are `<button>` elements that render meal name as visible text (acceptable), but no `aria-pressed` or `aria-disabled` is set to reflect the logged/unlogged state.

**Suggested Fixes:**
```tsx
// DateSelector — chevrons
<Button aria-label="Jour précédent" ...><ChevronLeft /></Button>
<Button aria-label="Jour suivant" ...><ChevronRight /></Button>

// MealSection — delete
<Button aria-label={`Supprimer ${entry.food_name}`} ...><Trash2 /></Button>

// TrackingInput — mic
<button aria-label={isListening ? "Arrêter l'écoute" : "Dicter un repas"}
        aria-pressed={isListening} ...>

// PlanValidation — meal rows
<button aria-pressed={logged} aria-label={`${meal.name} — ${logged ? 'déjà enregistré' : 'enregistrer'}`} ...>
```

## Cross-cutting Concerns

**Pattern: TypeScript strictness disabled**
Both `tsconfig.app.json` (`strict: false`, `strictNullChecks: false`) and `tsconfig.json` (`noImplicitAny: false`) disable the most important safety checks. This is why `jsonError.message` on line 238 of `api.ts` compiles despite `jsonError` being typed as `unknown` in strict mode. The project standard in `CLAUDE.md` states "No `any` in TypeScript. Strict mode." — the tsconfig contradicts this. Not flagging as a per-file issue (it's project-wide config), but this is the root cause of several type-safety gaps.

**Pattern: Repeat issue from previous review**
Issue #6 (dead streaming variables) is an extension of Issue #4 from the previous review (`review-20260304-100652.md`), which flagged the `API_BASE_URL` fragility in `api.ts`. The streaming block in `sendMessage` continues to accumulate dead code from incremental edits. The function is now 190 lines long and would benefit from extraction into smaller named helpers.

## Recommendations

- **Issue #2**: Replace `key={idx}` with `key={meal.name}` in `PlanValidation`.

### Process Improvement
Consider enabling `strictNullChecks: true` in `tsconfig.app.json`. The dead variable `title: string | undefined` and the `jsonError.message` access on an `unknown` would both be caught by the compiler rather than a code review.
