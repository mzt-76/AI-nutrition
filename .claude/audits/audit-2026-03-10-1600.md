## Cross-Cutting Patterns

### Pattern 1: Inconsistent UUID/Input Validation at API Boundaries
**Agents**: B (security) + D (standards)
Multiple API endpoints accept `user_id`, `recipe_id`, `request_id` as bare strings without UUID format validation. Some endpoints validate (e.g., `/api/recipes/{recipe_id}` calls `_validate_uuid()`), but most don't. Pydantic models inconsistently apply UUID patterns — `AgentRequest.user_id` has regex validation, but `DailyLogCreate`, `FavoriteCreate`, `ShoppingListCreate` don't. This is a systemic defense-in-depth gap.

### Pattern 2: Eval Files Missing Rule 11 Compliance (TEST_USER_PROFILE + user_id)
**Agent**: D (standards)
All 5 inspected eval files define personas inline in docstrings but don't export `TEST_USER_PROFILE` as a module-level constant. Additionally, evals testing DB-dependent skills (meal-planning, weekly-coaching) don't pass `user_id` to `create_agent_deps()`, causing silent fallback to improvised data. This undermines eval validity.

### Pattern 3: Type Safety Gaps in Backend Data Flows
**Agents**: A (backend) + B (security)
Several core modules have mypy errors from imprecise types: `feedback_extraction.py` assigns tuples to `str | list | None` variables, `tools.py` has 18 type misassignments, `api.py` has 28 union-attr errors around the streaming code. These aren't just linter noise — they indicate real type confusion that could cause runtime errors.

## Detailed Findings

### Critical

#### C-1: Unsafe type casting of component props without Zod validation
- **category**: correctness
- **file**: frontend/src/components/chat/MessageItem.tsx
- **lines**: 42-54
- **agent**: C (frontend)
- **what**: `handleMealClick` extracts props from UIComponentBlock using `as` casts (e.g., `p.recipe_name as string`) without Zod validation. ComponentRenderer validates props at render time, but MessageItem's callback uses the original unvalidated `comp.props`, bypassing the safety contract.
- **why**: Malformed LLM-generated data could reach RecipeDetailDrawer, causing runtime errors or unexpected behavior.
- **evidence**: Lines 42-54 cast `comp.props` directly with `as` instead of using the same Zod schemas that ComponentRenderer applies.
- **suggested_fix**: Validate props in handleMealClick using the MealCard Zod schema before casting, or store validated props from ComponentRenderer and pass them through.
- **confidence**: verified

### High

#### H-1: Conversation title not sanitized before database storage (XSS vector)
- **category**: security
- **file**: src/api.py
- **lines**: 1381-1387
- **agent**: B (security)
- **what**: `generate_conversation_title()` output is stored directly to the `conversations.title` field via `update_conversation_title()` with only `.strip()`. The LLM output is not sanitized, creating a stored XSS vector.
- **why**: Defense-in-depth requires backend sanitization. An attacker could craft a query causing the LLM to output HTML payloads persisted in the DB.
- **evidence**: Line 1385 calls `await update_conversation_title(supabase, session_id, conversation_title)` without `sanitize_user_text()`.
- **suggested_fix**: Add `conversation_title = sanitize_user_text(conversation_title, 100, "conversation_title")` before line 1385.
- **confidence**: verified

#### H-2: Unvalidated recipe_id in /api/favorites/check endpoint
- **category**: security
- **file**: src/api.py
- **lines**: 916-940
- **agent**: B (security)
- **what**: The `recipe_id` query parameter is not validated as UUID before Supabase query. Compare: `/api/recipes/{recipe_id}` correctly calls `_validate_uuid()`.
- **why**: Invalid UUIDs should be rejected at API layer (400) before reaching the database.
- **evidence**: Line 919 `recipe_id: str` with no validation. Line 930 uses it directly in `.eq()`.
- **suggested_fix**: Add `_validate_uuid(recipe_id)` after auth check, before line 926.
- **confidence**: verified

#### H-3: Type mismatch in feedback_extraction.py — tuples assigned to str | list | None
- **category**: correctness
- **file**: src/nutrition/feedback_extraction.py
- **lines**: 259, 293-324, 340-346
- **agent**: A (backend)
- **what**: `extract_feedback_from_text()` assigns tuples like `("high", 0.7)` to `energy_level`/`hunger_level` fields typed as `str | list[Any] | None`. Later code calls `.append()` on fields that may be `str` or `None`.
- **why**: Type mismatch causes 11 mypy errors and potential runtime AttributeError. Used by weekly-coaching skill.
- **evidence**: mypy: "Incompatible types in assignment (expression has type `tuple[str, float]`, target has type `str | list[Any] | None`)"
- **suggested_fix**: Use TypedDict or explicit types: `energy_level: tuple[str, float] | None`, initialize lists explicitly as `mood_indicators: list[str] = []`.
- **confidence**: verified

#### H-4: Implicit Optional in macro_adjustments.py
- **category**: correctness
- **file**: src/nutrition/macro_adjustments.py
- **lines**: 262, 269
- **agent**: A (backend)
- **what**: `user_allergens: list[str] = None` violates PEP 484 strict typing. Also `Collection[str]` indexing is not type-safe.
- **why**: mypy strict mode rejects this. Could mask bugs if function is called with unexpected types.
- **suggested_fix**: Change to `user_allergens: list[str] | None = None`.
- **confidence**: verified

#### H-5: Eval files missing TEST_USER_PROFILE constant (rule 11) — 5 files

#### H-6: Evals for DB-dependent skills missing user_id — 4 files
