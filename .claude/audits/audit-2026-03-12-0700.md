## Cross-Cutting Patterns

### Pattern 1: Service Key Architecture (5 findings)
The backend uses `SUPABASE_SERVICE_KEY` for ALL operations, bypassing RLS. This is the root cause of 5 interconnected findings: RLS bypass (S1), delete policy conflicts (S7), recipe INSERT policy mismatch (DB2), service key in JWT verification (S6), and rate limiter fail-open compounding with no DB-level defense (B20). **Fix as a single architectural change**: create user-scoped Supabase clients per request.

### Pattern 2: Rule 7b Constants Centralization (5 findings)
Hardcoded tunable parameters across skill scripts: `max_tokens=1500` (image_analysis), `RECIPE_MODEL` (2 files), `MIN_SIMILARITY=0.5`, embedding model name. All should be in `constants.py` or env vars.

### Pattern 3: Input Sanitization Gaps (2 high findings)
`sanitize_user_text()` exists and is used for agent queries and food search, but ALL CRUD endpoints skip it: `food_name` (daily log), `recipe.name`, `recipe.instructions`, `ingredient[].name`. The recipe fields are especially dangerous — shared table, cross-user attack vector.

## Detailed Findings

### Critical

#### C1: Service_role key bypasses all RLS policies
- **Found by**: security (S1), cross-validated by all 3 reviewers
- **File**: `src/clients.py:123-135`
- **What**: `get_async_supabase_client()` uses `SUPABASE_SERVICE_KEY`. All 60+ RLS policies are bypassed. Application-layer auth checks are the only defense.
- **Fix**: Create user-scoped client per request using JWT. Keep service key for admin/system operations only.

#### C2: `execute_custom_sql()` — SECURITY DEFINER with raw SQL concatenation
- **Found by**: database audit (DB1)
- **File**: `sql/0-all-tables.sql:576-594`
- **What**: `EXECUTE 'SELECT ... FROM (' || sql_query || ') t'` with SECURITY DEFINER. Revoked from PUBLIC but callable by service_role. No app code uses it.
- **Fix**: `DROP FUNCTION IF EXISTS public.execute_custom_sql(text);`

#### C3: `result.data` instead of `result.output` (pydantic-ai API crash)
- **Found by**: standards (ST1, ST2), cross-validated by backend
- **Files**: `src/agent.py:550`, `src/streamlit_ui.py:234`
- **What**: Deprecated pydantic-ai 0.0.x API. Both crash with `AttributeError`. Both files excluded from Docker, but still user-facing bugs in local dev.
- **Fix**: Change `.data` → `.output` in both files.

#### C4: Floor-pinned dependencies allow silent breaking upgrades
- **Found by**: standards (ST3), docker (D13), cross-validated by backend
- **File**: `requirements.txt`
- **What**: All 17 deps use `>=` (e.g., `pydantic-ai>=1.39.0`). Each Docker build could pull breaking versions. This is exactly how C3 happened historically.
- **Fix**: Pin critical deps with `==`. Generate `requirements.lock` from known-good env.

### High

#### H1: Missing `sanitize_user_text()` on RecipeCreate fields (cross-user XSS + prompt injection)
- **Found by**: security (S12), cross-validated by backend + frontend
- **File**: `src/api.py:976-1026`
- **What**: `name` (200 chars), `instructions` (5000 chars), `ingredients[].name` — all unsanitized. Recipes table is shared across ALL users. Malicious recipe content affects other users' meal plans.
- **Fix**: Apply `sanitize_user_text()` to all text fields before DB write.

#### H2: Missing `sanitize_user_text()` on DailyLogCreate food_name
- **Found by**: security (S11)
- **File**: `src/api.py:721-735`
- **What**: `food_name` written directly to DB, rendered in frontend. Stored XSS + prompt injection risk.
- **Fix**: `body.food_name = sanitize_user_text(body.food_name, 200, "daily_log_food_name")`

#### H3: Missing UUID validation on 7 GET endpoints
- **Found by**: security (S2), cross-validated by frontend
- **File**: `src/api.py` (7 endpoints)
- **What**: `user_id` as raw query param without `_validate_uuid()`. POST bodies validate via Pydantic, creating inconsistency.
- **Fix**: Add `_validate_uuid(user_id)` to all 7 endpoints, or create a FastAPI dependency.

#### H4: Nginx security headers silently dropped in all location blocks
- **Found by**: security (S3) + frontend (F1) + docker (D10) — triple-validated
- **File**: `frontend/nginx.conf:39-77`
- **What**: X-Frame-Options, X-Content-Type-Options, X-XSS-Protection, Referrer-Policy dropped in 5 location blocks due to `add_header` inheritance.
- **Fix**: Repeat security headers in every location block or use an `include` snippet.

#### H5: Missing CSP and HSTS headers in nginx
- **Found by**: security (S4) + frontend (F2)
- **File**: `frontend/nginx.conf`
- **What**: Backend API has CSP/HSTS but the user-facing frontend nginx does not.
- **Fix**: Add CSP and HSTS to server block (with correct connect-src for Supabase).

#### H6: Delete endpoints conflict with "Deny delete" RLS policies
- **Found by**: security (S7), compounds with C1
- **Files**: `src/api.py:596-662`, `sql/0-all-tables.sql`
- **What**: API has DELETE endpoints but RLS has `USING (false)`. Works only via service key. Fixing C1 breaks all deletes.
- **Fix**: Replace deny-delete policies with user-scoped delete policies for affected tables.

#### H7: Recipes RLS only allows admin INSERT but API allows any user
- **Found by**: database (DB2), compounds with C1
- **What**: RLS: `WITH CHECK (is_admin())`. API: any authenticated user can insert. Another C1 dependency.
- **Fix**: Add authenticated user INSERT policy on recipes table.

#### H8: Dev dependencies (~200MB) in production Docker image
- **Found by**: standards (ST4), docker (D12)
- **File**: `requirements.txt:33-35`
- **What**: streamlit (~150MB), rich, Pillow installed but their code is excluded from Docker.
- **Fix**: Create `requirements-prod.txt` without streamlit/rich. Use in Dockerfile.

#### H9: `image_url` not validated for scheme (SSRF risk)
- **Found by**: backend (B16), elevated to HIGH by security cross-review
- **File**: `skills/body-analyzing/scripts/image_analysis.py:26-27`
- **What**: URL passed to OpenAI API without scheme validation. Privacy-sensitive health image context.
- **Fix**: Validate `image_url.startswith("https://")`.

#### H10: Rate limiter fails open on DB errors
- **Found by**: backend (B20), elevated to HIGH by security cross-review
- **File**: `src/db_utils.py`
- **What**: Returns `True` (allowed) on DB errors. Compounds with C1 — if DB is down and rate limits are disabled, no defense.
- **Fix**: Implement in-memory fallback rate limiter or fail closed.

#### H11: `spec.loader` null check missing in dynamic imports
- **Found by**: standards (ST6) + backend (B11) — in `agent.py:149` and `generate_week_plan.py:93`
- **What**: `spec_from_file_location()` can return None. Crashes with confusing `AttributeError`.
- **Fix**: Add `if spec is None or spec.loader is None: raise ImportError(...)`.

#### H12: Macro tolerance inconsistency between validators
- **Found by**: backend (B6), elevated to HIGH by standards cross-review
- **File**: `src/nutrition/validators.py:387-388`
- **What**: Generic validator uses flat 10% tolerance. Day plan pipeline uses per-macro tolerances from constants.py. Inconsistent validation.
- **Fix**: Update `validate_daily_macros` to use per-macro tolerances from constants.py.

- **File upload validation** — MIME whitelist, size checks
