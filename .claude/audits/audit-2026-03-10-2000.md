**Overall Health:** Good. The codebase follows its own CLAUDE.md rules well. The 6-tool agent boundary is respected, skill scripts import from `src/nutrition/` rather than reimplementing, safety constraints use constants, and scripts/ are LLM-free. The main concerns are: (1) macro tolerance constants not centralized in `constants.py`, (2) legacy `typing.Dict/List/Optional` usage in several files, and (3) missing RLS migration files for newer tables.

**Top Concern:** `MACRO_TOLERANCE_*` constants defined in `validators.py` instead of `constants.py`, violating CLAUDE.md rule 7b which states all tunable constants must be centralized there.

## Cross-Cutting Patterns

1. **Legacy typing imports**: Files like `skill_loader.py`, `calculations.py`, `macro_adjustments.py`, and RAG_Pipeline modules use `Dict`, `List`, `Optional` from `typing` instead of modern `dict`, `list`, `X | None` syntax. This is inconsistent with the rest of the codebase which uses modern syntax.

2. **Constants not fully centralized**: `MACRO_TOLERANCE_*` in `validators.py` and `FAT_PCT_OF_TOTAL`, caloric values per gram in `calculations.py` are inline rather than in `constants.py`.

3. **API file is very large**: `api.py` at 1477 lines contains all CRUD endpoints, streaming logic, auth, and middleware. While not a rule violation, it could benefit from splitting.

## Detailed Findings

### HIGH

### Finding 1
- **severity**: high
- **category**: standards
- **file**: src/nutrition/validators.py
- **lines**: 119-122
- **title**: MACRO_TOLERANCE constants not in constants.py (rule 7b violation)
- **what**: Four tunable pipeline constants (`MACRO_TOLERANCE_PROTEIN`, `MACRO_TOLERANCE_FAT`, `MACRO_TOLERANCE_CALORIES`, `MACRO_TOLERANCE_CARBS`) are defined in `validators.py` instead of `constants.py`. Rule 7b says "all pipeline-tunable parameters... are centralized in one file."
- **why**: Having tunable constants scattered across files defeats the purpose of `constants.py` as a single tuning knob. Someone modifying tolerances won't find them in the expected location.
- **evidence**: `validators.py:119-122` defines them, `generate_day_plan.py:31-34` imports them from validators.
- **suggested_fix**: Move the 4 constants to `src/nutrition/constants.py` under a new section. Update imports in `validators.py` and `generate_day_plan.py`.
- **confidence**: verified

### Finding 2
- **severity**: high
- **category**: security
- **file**: sql/
- **lines**: N/A
- **title**: Missing RLS migration files for newer tables
- **what**: No SQL migration files exist for RLS on `conversations`, `messages`, `daily_food_log`, `favorite_recipes`, or `shopping_lists` tables. The project states "all RLS-enabled" but the migration files only cover `meal_plans`, `weekly_feedback`, and `user_learning_profile`. If RLS was applied via console, it's not version-controlled and could be lost on a fresh deployment.
- **why**: Without version-controlled RLS migrations, a database reset or new deployment could expose all user data. The frontend queries `conversations` and `messages` directly via Supabase JS client with the anon key, relying on RLS for access control.
- **evidence**: `sql/6_rls_per_user_tables.sql` only covers 3 tables. No other SQL file mentions conversations, messages, daily_food_log, favorite_recipes, or shopping_lists.
- **suggested_fix**: Create `sql/14_rls_conversations_messages.sql` and `sql/15_rls_food_favorites_shopping.sql` with the appropriate RLS policies, matching the pattern in migration 6.
- **confidence**: probable

### Finding 3
- **severity**: high
- **category**: modularity
- **file**: src/api.py
- **lines**: 1-1477
- **title**: API module is 1477 lines with mixed concerns
- **what**: `api.py` contains JWT auth, middleware, CORS config, request/response models, all CRUD endpoints (conversations, food log, favorites, recipes, shopping lists, meal plans, profile recalculation), streaming logic, and error handling in a single file. This makes it harder to navigate and test individual concerns.
- **why**: While not explicitly prohibited by CLAUDE.md rules, the file handles 7+ resource domains. Adding new endpoints increases coupling risk and review difficulty.
- **evidence**: File has 1477 lines, 20+ endpoint functions, 11 Pydantic models, auth logic, and streaming logic.
- **suggested_fix**: Extract models to `src/api_models.py`, auth to `src/api_auth.py`, and consider splitting CRUD endpoints into route modules (e.g., `src/routes/food_log.py`). Keep `api.py` as the app factory.
- **confidence**: verified

### MEDIUM

### Finding 4
- **severity**: medium
- **category**: standards
- **file**: src/skill_loader.py, src/nutrition/calculations.py, src/nutrition/macro_adjustments.py
- **lines**: various
- **title**: Legacy typing imports (Dict, List, Optional) instead of modern builtins
- **what**: Several files import and use `Dict`, `List`, `Optional`, `Tuple` from `typing` instead of the modern `dict`, `list`, `X | None` syntax. This is inconsistent with the rest of the codebase (e.g., `agent.py`, `tools.py`, `api.py`) which uses modern syntax.
- **why**: Inconsistency. Python 3.11+ supports `dict[str, int]` natively. The `from typing import Dict` pattern is deprecated since Python 3.9.
- **evidence**: 21 total occurrences across 5 files: `skill_loader.py` (4), `calculations.py` (3), `macro_adjustments.py` (2 + many in function signatures), RAG_Pipeline files (12).
- **suggested_fix**: Replace `Dict[` with `dict[`, `List[` with `list[`, `Optional[X]` with `X | None`, `Tuple[` with `tuple[`. Remove unused typing imports.
- **confidence**: verified

### Finding 5
- **severity**: medium
- **category**: standards
- **file**: src/nutrition/calculations.py
- **lines**: 322-334
- **title**: Inline magic constants not in constants.py
- **what**: `FAT_PCT_OF_TOTAL` dict (lines 329-334), caloric values `PROTEIN_KCAL=4`, `CARB_KCAL=4`, `FAT_KCAL=9` (lines 323-325) are defined inline in `calculate_macros()`. These are tunable pipeline parameters per rule 7b.
- **why**: If someone wants to adjust fat percentage by goal or caloric conversion factors, they'd need to modify `calculations.py` instead of the single-source `constants.py`.
- **evidence**: `calculate_macros()` defines `FAT_PCT_OF_TOTAL` as a local variable. While caloric values are physical constants (not truly tunable), the fat percentages are goal-dependent design choices.
- **suggested_fix**: Move `FAT_PCT_OF_TOTAL` to `constants.py`. Caloric values (4/4/9) are universal constants and could stay inline or move to constants.py with a comment noting they're physical constants.
- **confidence**: verified

### Finding 6
- **severity**: medium
- **category**: correctness
- **file**: src/nutrition/validators.py
- **lines**: 505
- **title**: Calorie tolerance uses carbs_tolerance instead of dedicated calorie tolerance
- **what**: In `validate_meal_plan_macros()`, calorie validation uses `carbs_tolerance` (line 505: `cal_lower = target_calories * (1 - carbs_tolerance)`). But the function has no explicit `calorie_tolerance` parameter. Meanwhile, `MACRO_TOLERANCE_CALORIES = 0.10` exists separately in the same file. The default `carbs_tolerance=0.10` happens to match, but the coupling is misleading.
- **why**: If `carbs_tolerance` is changed to a different value (it defaults to 0.10 but MACRO_TOLERANCE_CARBS is 0.20), calorie validation would silently change tolerance. The intent is unclear.
- **evidence**: Line 505: `cal_lower = target_calories * (1 - carbs_tolerance)` — uses carbs tolerance for calorie validation.
- **suggested_fix**: Add an explicit `calorie_tolerance` parameter to `validate_meal_plan_macros()` and use it for calorie validation instead of reusing `carbs_tolerance`.
- **confidence**: verified

### Finding 7
- **severity**: medium
- **category**: security
- **file**: frontend/src/lib/api.ts
- **lines**: 460-476
- **title**: fetchMessages queries by session_id without user ownership validation
- **what**: `fetchMessages()` queries the `messages` table by `session_id` only. It accepts a `_user_id` parameter but doesn't use it (prefixed with underscore). If a user guesses or obtains another user's session_id, they could read their messages — unless RLS on the `messages` table restricts this. The session_id format (`{user_id}~{random_10_chars}`) provides some obscurity but is not cryptographic.
- **why**: Defense in depth. Even with RLS, the function signature suggests user_id should be used for validation but explicitly ignores it.
- **evidence**: `fetchMessages(session_id: string, _user_id: string)` — underscore prefix indicates intentionally unused parameter.
- **suggested_fix**: If RLS handles this (which it should), remove the `_user_id` parameter entirely to avoid confusion. If RLS doesn't handle it, add `.eq('user_id', user_id)` via a join or filter. Verify RLS policy exists for `messages` table.
- **confidence**: needs-testing

### Finding 8
- **severity**: medium
- **category**: modularity
- **file**: src/nutrition/validators.py
- **lines**: 1-847
- **title**: validators.py contains scoring logic and macro tolerances beyond validation
- **what**: `validators.py` (847 lines) contains `find_worst_meal()` scoring logic and `MACRO_TOLERANCE_*` constants which are not strictly validation concerns. The file mixes pure validation (allergen checks, structure validation) with scoring (find worst meal) and pipeline constants.
- **why**: Makes the file harder to navigate. `find_worst_meal` is a scoring/selection function, not a validator. Constants should be in `constants.py`.
- **evidence**: `find_worst_meal()` at lines 343-393 is a scoring function used for meal replacement decisions, not validation.
- **suggested_fix**: Move `find_worst_meal()` to `src/nutrition/meal_planning.py` or a new `src/nutrition/scoring.py`. Move MACRO_TOLERANCE_* to `constants.py`.
- **confidence**: verified

### Finding 9
- **severity**: medium
- **category**: efficiency
- **file**: src/nutrition/validators.py
- **lines**: 621
- **title**: Meal plan structure validation only checks first meal per day
- **what**: `validate_meal_plan_structure()` validates ingredient and nutrition fields only on `meals[0]` (line 623: `meal = meals[0]`), skipping all subsequent meals. A malformed meal at index 1+ would pass validation.
- **why**: A corrupted later meal (e.g., missing `ingredients` or `nutrition` field) would not be caught, potentially causing downstream errors in macro totaling or display.
- **evidence**: Comment on line 621: `# Validate each meal structure (sample first meal only for performance)`.
- **suggested_fix**: Iterate over all meals, not just the first. The performance impact of checking 3-5 meals per day (21-35 total for a week plan) is negligible.
- **confidence**: verified

### Finding 10
- **severity**: medium
- **category**: standards
- **file**: src/RAG_Pipeline/
- **lines**: multiple files
- **title**: RAG_Pipeline has its own venv committed to the repo
- **what**: `src/RAG_Pipeline/venv/` is committed to the repository, containing third-party packages. This inflates the repo size (287K+ total Python LOC includes venv packages) and makes `find`/`grep` operations slower.
- **why**: Virtual environments should be gitignored. They're platform-specific and can be recreated from requirements.txt.
- **evidence**: `find src/ -name '*.py'` returns files like `src/RAG_Pipeline/venv/lib/python3.12/site-packages/aiohttp/client.py`.
- **suggested_fix**: Add `src/RAG_Pipeline/venv/` to `.gitignore` and remove it from git tracking with `git rm -r --cached src/RAG_Pipeline/venv/`.
- **confidence**: verified

### Finding 11
- **severity**: low
- **category**: simplicity
- **file**: src/nutrition/validators.py
- **lines**: 192-213, 243-302
- **title**: Duplicated allergen matching logic between matches_allergen and validate_allergens
- **what**: `matches_allergen()` and `validate_allergens()` both implement allergen keyword matching with false positive checks. `validate_allergens()` could call `matches_allergen()` internally instead of reimplementing the same logic.
- **why**: DRY violation. If allergen matching logic changes, both functions need updating.
- **evidence**: Both functions normalize allergens, check `ALLERGEN_FALSE_POSITIVES`, iterate `ALLERGEN_FAMILIES`, and do keyword matching.
- **suggested_fix**: Refactor `validate_allergens()` to call `matches_allergen()` for each ingredient, replacing its inline matching logic.
- **confidence**: verified
