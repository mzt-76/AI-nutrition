## Cross-Cutting Patterns

### Pattern 1: Inconsistent `sanitize_user_text()` application
`sanitize_user_text()` is called too late in the agent endpoint (inside the streaming generator, after DB writes and mem0/title-agent calls) and not called at all in `/api/food-search`. This creates a systematic input-validation gap across 3 entry points. **Found by**: security + backend agents, cross-validated by all 4.

### Pattern 2: Test/eval boundary violations (rule 11)
Real LLM tests live in `tests/` (2 files, 25 test methods), and deterministic mock-based tests live in `evals/` (2 files). The boundary is violated in both directions, making CI unreliable and hiding test coverage. **Found by**: standards agent, confirmed by all reviewers.

### Pattern 3: Dead code accumulation enabled by disabled compiler checks
`tsconfig.app.json` disables `noUnusedLocals` and `noUnusedParameters`, which masks dead exports (`setConversations`), unused schema fields (`message_data`), and style inconsistencies. On the Python side, re-export hacks (`MEAL_STRUCTURES`, `round_quantity_smart`) with `# noqa: F401` serve a similar purpose. **Found by**: standards + frontend agents, compound confirmed.

## Detailed Findings

### Critical

#### C1: Late sanitization ‚Äî raw user query persisted to DB and fed to title-agent before guard fires
- **file**: `src/api.py:395, 433, 451, 487, 1283`
- **agents**: security (F1) + backend (F10) ‚Äî cross-validated as compound
- **confidence**: verified
- **what**: `sanitize_user_text()` is only called inside `_stream_agent_response()` at line 1283, well after the raw `request.query` has been written to `messages` (line 433), `requests` (line 395), passed to `mem0_client.search` (line 451), and `generate_conversation_title` (line 499). Additionally, `/api/food-search` never calls `sanitize_user_text()` on its `q` parameter.
- **why**: A prompt-injection payload is persisted to the database and fed to the title-generation LLM (a cheaper model with no system-prompt hardening) before any guard fires. The title-agent injection is the highest-risk vector.
- **fix**: Sanitize `request.query` at the top of `agent_endpoint`, before any DB writes. Apply `sanitize_user_text()` to `/api/food-search` `q` parameter. Remove the redundant sanitization inside `_stream_agent_response`.

#### C2: `POST /api/recipes` has no macro validation and misleading `source` label ~~(downgraded from critical to high after investigation)~~
- **file**: `src/api.py:947-1000`
- **agents**: security (F2) ‚Äî originally elevated to critical, **reassessed to high** after investigating the actual usage flow
- **confidence**: verified
- **what**: `POST /api/recipes` requires only `require_auth` (any valid JWT). The server-side Supabase client uses `SUPABASE_SERVICE_KEY` which bypasses the admin-only RLS INSERT policy. However, this endpoint is legitimately needed: it's called by `RecipeDetailDrawer` when a user favorites a meal from a generated plan. The meal data is already OFF-validated by the agent pipeline. The real risks are: (1) a direct API call with falsified macros, and (2) the `source: "user_validated"` label gives unearned trust to user-saved recipes.
- **why**: An attacker calling the API directly (not through the UI) could inject recipes with falsified macros. The `_passes_atwater_check()` function exists but is not applied here.
- **fix**: Add Atwater validation (`_passes_atwater_check`) before insert (reject if >30% discrepancy). Change `source` from `"user_validated"` to `"user_saved"`. No admin check needed ‚Äî the favorites flow requires this endpoint.

#### C3: Safety calorie minimum not enforced in agent‚Üískill path (rule 3 violation)
- **file**: `skills/nutrition-calculating/scripts/calculate_nutritional_needs.py:80-82`
- **agents**: backend (F9) ‚Äî elevated to critical by security reviewer, confirmed by standards
- **confidence**: verified
- **what**: The `calculate_nutritional_needs` skill script computes `target_calories` but does NOT enforce `MIN_CALORIES_WOMEN = 1200` / `MIN_CALORIES_MEN = 1500`. The safety floor only exists in `src/api.py:1201-1202` (REST endpoint). When the agent calls this skill directly (the primary use case), sub-minimum recommendations can be produced.
- **why**: CLAUDE.md rule 3: "Safety Constraints ‚Äî hardcoded, never bypass." A sedentary woman on weight loss could receive <1200 kcal recommendations through the chat agent.
- **fix**: After computing `target_calories`, add:
```python
from src.nutrition.constants import MIN_CALORIES_WOMEN, MIN_CALORIES_MEN
min_cal = MIN_CALORIES_WOMEN if gender == "female" else MIN_CALORIES_MEN
target_calories = max(target_calories, min_cal)
```

#### H2: `requests` table has no RLS policy ‚Äî user health queries readable by any authenticated user
