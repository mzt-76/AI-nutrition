# Feature: PWA → TWA (APK Android)

The following plan should be complete, but validate documentation and codebase patterns before implementing.

## Feature Description

Transformer l'application web (déjà configurée en PWA) en APK Android installable via TWA (Trusted Web Activity). L'APK wrappe le site web existant dans une coque Android native qui lance Chrome en mode plein écran, sans barre de navigation. La PWA est déjà fonctionnelle (manifest, service worker, icônes). Il reste à configurer Digital Asset Links, générer l'APK avec Bubblewrap, et distribuer.

## User Story

As a **membre de la famille/ami du créateur**
I want to **installer l'app NutriAI sur mon téléphone Android comme une vraie application**
So that **j'y accède facilement depuis mon écran d'accueil sans ouvrir un navigateur**

## Problem Statement

L'application est déployée en production sur Render et fonctionne en PWA, mais les utilisateurs non-techniques (famille, amis) ne savent pas comment "installer" une PWA depuis le navigateur. Un APK partagé via WhatsApp/Telegram est plus naturel et accessible.

## Solution Statement

Utiliser Bubblewrap (outil Google officiel) pour wrapper la PWA dans un TWA Android. Le TWA lance Chrome en mode standalone pour afficher le site web — aucun code natif nécessaire. La vérification Digital Asset Links prouve que l'APK et le site appartiennent au même développeur, ce qui supprime la barre d'URL Chrome.

## Feature Metadata

**Feature Type**: New Capability (distribution)
**Estimated Complexity**: Medium
**Primary Systems Affected**: Frontend (public assets), Render deployment, Android toolchain (externe)
**Dependencies**: Bubblewrap CLI, JDK 17, Android SDK (auto-installés par Bubblewrap)

### Key Technical Facts

- **Bubblewrap on WSL2**: Works. Node.js CLI + Android SDK Linux builds. ADB device access from WSL2 is tricky → copy APK to Windows-accessible path and use `adb.exe` from PowerShell, or direct file transfer to phone.
- **JDK**: Let Bubblewrap download its own (JDK 17 Temurin). Don't use system JDK.
- **Android SDK**: Auto-downloaded by Bubblewrap on first run. No Android Studio needed.
- **Keystore passwords**: Key password and store password **MUST be identical** or build fails.
- **Play App Signing gotcha**: If publishing to Play Store, Google re-signs with their key. Need TWO SHA256 fingerprints in assetlinks.json (upload key + Google signing key from Play Console > App Integrity).
- **manifest.json is already valid** for TWA — has `standalone`, 192+512 icons, theme_color, background_color.

## STEP-BY-STEP TASKS

### Task 1: CREATE `frontend/public/.well-known/assetlinks.json`

- **IMPLEMENT**: Créer le fichier Digital Asset Links avec un placeholder fingerprint
  ```json
  [
    {
      "relation": ["delegate_permission/common.handle_all_urls"],
      "target": {
        "namespace": "android_app",
        "package_name": "com.ainutrition.app",
        "sha256_cert_fingerprints": [
          "PLACEHOLDER:REPLACE:WITH:REAL:SHA256:FINGERPRINT"
        ]
      }
    }
  ]
  ```
- **GOTCHA**: Le `package_name` doit être unique et en format reverse-domain. Utiliser `com.ainutrition.app`.
- **VALIDATE**: `cat frontend/public/.well-known/assetlinks.json | python -m json.tool`

### Task 2: VERIFY Render sert `.well-known/` correctement

- **IMPLEMENT**: Le rewrite `/* → /index.html` dans render.yaml pourrait intercepter `.well-known/assetlinks.json`. Render static sites servent les fichiers statiques en priorité avant les rewrites, donc ça devrait marcher. Mais il faut vérifier après déploiement.
- **VALIDATE**: Après push et déploiement sur Render :
  ```bash
  curl -I https://ai-nutrition-frontend-78p7.onrender.com/.well-known/assetlinks.json
  # Doit retourner 200 avec Content-Type: application/json
  ```
- **FALLBACK**: Si 404, ajouter une route spécifique dans nginx.conf :
  ```nginx
  location /.well-known/ {
      default_type application/json;
  }
  ```
  Mais note : le nginx.conf est utilisé uniquement dans le Docker deployment, pas dans le Render static site. Sur Render static, les fichiers dans `dist/` sont servis directement — `.well-known/` devrait être copié par le build Vite depuis `public/`.

### Task 3: INSTALL Bubblewrap CLI

- **IMPLEMENT**:
  ```bash
  npm install -g @bubblewrap/cli
  ```
- **GOTCHA**: Sur WSL2, utiliser Node.js installé dans WSL (pas la version Windows).
- **VALIDATE**: `bubblewrap --version`

### Task 4: INIT le projet TWA avec Bubblewrap

- **IMPLEMENT**:
  ```bash
  mkdir -p twa && cd twa
  bubblewrap init --manifest=https://ai-nutrition-frontend-78p7.onrender.com/manifest.json
  ```
- **PROMPTS** (valeurs à entrer pendant l'init interactif) :
  | Prompt | Valeur |
  |--------|--------|
  | Application name | Assistant Nutrition IA |
  | Short name | NutriAI |
  | Application ID | com.ainutrition.app |
  | Display mode | standalone |
  | Orientation | portrait |
  | Status bar color | #0a0c10 |
  | Splash screen color | #0a0c10 |
  | Icon URL | (auto-filled from manifest) |
  | Keystore location | ./android.keystore |
  | Key alias | nutriai |
  | Keystore password | (choisir un mot de passe — **GARDER PRÉCIEUSEMENT**) |
  | Key password | (**MÊME** mot de passe que le keystore) |

- **GOTCHA**: Bubblewrap demandera de télécharger JDK et Android SDK — répondre OUI. Ça prend quelques minutes (~500 MB).
- **VALIDATE**: `ls twa/` devrait contenir `build.gradle`, `app/`, `gradle/`, etc.

### Task 5: EXTRACT SHA256 fingerprint et UPDATE assetlinks.json

- **IMPLEMENT**:
  ```bash
  cd twa
  keytool -list -v -keystore android.keystore -alias nutriai
  # Copier la ligne SHA256 (format AA:BB:CC:...)
  ```
  Ou utiliser Bubblewrap directement :
  ```bash
  bubblewrap fingerprint generateAssetLinks
  ```
- **UPDATE**: Remplacer le placeholder dans `frontend/public/.well-known/assetlinks.json` par le vrai fingerprint.
- **VALIDATE**: `python -m json.tool frontend/public/.well-known/assetlinks.json` (JSON valide)

### Task 6: DEPLOY assetlinks.json en production

- **IMPLEMENT**: Commit + push les changements (assetlinks.json avec le vrai fingerprint). Render redéploie automatiquement.
- **VALIDATE**:
  ```bash
  curl https://ai-nutrition-frontend-78p7.onrender.com/.well-known/assetlinks.json
  # Doit retourner le JSON avec le bon fingerprint
  ```
- **VALIDATE** (Google tool):
  ```
  https://developers.google.com/digital-asset-links/tools/generator
  → Hosting site domain: ai-nutrition-frontend-78p7.onrender.com
  → App package name: com.ainutrition.app
  → App package fingerprint: <ton SHA256>
  → Vérifier que ça passe
  ```

### Task 7: BUILD l'APK

- **IMPLEMENT**:
  ```bash
  cd twa
  bubblewrap build
  ```
- **OUTPUT**: Deux fichiers générés :
  - `app-release-signed.apk` — pour installation directe
  - `app-release-bundle.aab` — pour Play Store (si besoin plus tard)
- **VALIDATE**: `ls -la twa/app-release-signed.apk` (doit exister, taille ~3-10 MB)

### Task 8: TEST sur device Android

- **IMPLEMENT** (Option la plus simple — transfert direct) :
  ```bash

  cp twa/app-release-signed.apk /mnt/c/Users/meuze/Downloads/NutriAI.apk
  ```
  Puis envoyer `NutriAI.apk` au téléphone (USB, email, Google Drive, ou WhatsApp à soi-même).

- **IMPLEMENT** (Option ADB si device connecté) :
  ```bash
  # Depuis PowerShell Windows (pas WSL2) :
  adb install C:\Users\meuze\Downloads\NutriAI.apk
  ```

## NOTES

### Décisions de design

1. **Distribution directe APK** (pas Play Store) — plus rapide, pas de frais ni d'attente de review. Le Play Store est une option future si l'app est validée auprès des premiers utilisateurs.

2. **Pas de mode offline avancé** — L'app shell cache déjà (HTML/CSS/JS), les API calls restent NetworkOnly. Suffisant pour un MVP. L'offline data caching est un enhancement futur.

3. **Package name `com.ainutrition.app`** — Simple, mémorable. Si on publie sur le Play Store plus tard, c'est un bon ID.

4. **Répertoire `twa/` gitignored** — Les artifacts Android sont volumineux et machine-spécifiques. Seul `assetlinks.json` et le script helper sont versionnés.

### Risques identifiés

| Risque | Probabilité | Impact | Mitigation |
|--------|-------------|--------|------------|
| OAuth Google échoue dans WebView | Moyen | Bloquant | Tester tôt. Les TWA utilisent Chrome (pas un WebView classique), donc OAuth devrait marcher. Si problème : vérifier les Redirect URLs dans Supabase dashboard. |
| assetlinks.json non servi par Render | Faible | Bloquant | Render static sert les fichiers de `dist/` en priorité. Le build Vite copie `public/.well-known/` vers `dist/.well-known/`. |
| Barre Chrome visible (DAL verification fail) | Moyen | UX dégradée | Vérifier fingerprint avec Google validator. Parfois Chrome met 24-48h à vérifier. Vider le cache Chrome. |
| ADB non accessible depuis WSL2 | Moyen | Ralentissant | Contournement : copier l'APK vers Windows et installer directement sur le téléphone via file transfer. |
| Keystore perdu | Faible | Critique | Sauvegarder immédiatement dans 1Password / Google Drive privé. Sans le keystore, impossible de publier des mises à jour signées. |

### Pour le Play Store (futur, optionnel)

Si on veut publier sur le Play Store plus tard :
1. Créer un compte Google Play Developer (25$ one-time)
2. Upload l'AAB (pas l'APK) sur Play Console
3. Ajouter le SHA256 de la clé de signing Google (Play Console > App Integrity) à `assetlinks.json`
4. Préparer : screenshots, description FR, icône haute résolution (512x512)
5. Utiliser l'Internal Testing track d'abord (100 testeurs, disponible en minutes)
